# local-ai-proxy

Exposes your **local Ollama** (or any custom server) to the internet through a
VPS using [FRP](https://github.com/fatedier/frp) — no port-forwarding on your
router required.

```
Internet  ──►  VPS (frps)  ──tunnel──►  Home PC (frpc + Ollama)
```

---

```

│   ├── docker-compose.yml  (auto-generated by setup.sh)

│   └── .env.example
└── local/                ← Run on your home PC
    ├── docker-compose.yml
    ├── frpc.ini            (auto-generated by setup.sh)
    └── .env.example
```

## Quick start

### 1 — Generate a shared secret

```bash
openssl rand -hex 32
# e.g. a3f9c2d1e8b7…  ← use this as FRP_TOKEN everywhere
```

### 2 — Set up the VPS

```bash
# Upload the vps/ folder to your server, then:
cd vps/

cp .env.example .env
# Edit .env — set FRP_TOKEN and DASHBOARD_PWD

# Substitute placeholders in frps.ini
sed -i "s/{{ FRP_TOKEN }}/$FRP_TOKEN/g"       frps.ini
sed -i "s/{{ DASHBOARD_PWD }}/$DASHBOARD_PWD/g" frps.ini

docker compose up -d
```

Open firewall ports on the VPS: **7000** (frp control), **11434** (Ollama API),
**7500** (dashboard, optional).

### 3 — Set up your local PC

```bash
cd local/

cp .env.example .env
# Edit .env — set VPS_HOST and FRP_TOKEN (same token as on the VPS)

# Substitute placeholders in frpc.ini
sed -i "s/{{ VPS_HOST }}/$VPS_HOST/g"   frpc.ini
sed -i "s/{{ FRP_TOKEN }}/$FRP_TOKEN/g" frpc.ini

docker compose up -d

# Pull a model (first run only)
docker exec -it local-ollama-1 ollama pull llama3
```

### 4 — Test it

```bash
# From any machine on the internet:
curl http://<YOUR_VPS_IP>:11434/api/tags

# Chat completion:
curl http://<YOUR_VPS_IP>:11434/api/chat \
  -d '{"model":"llama3","messages":[{"role":"user","content":"Hello!"}]}'
```

## GPU support (NVIDIA)

Uncomment the `deploy.resources` block in `local/docker-compose.yml` and make
sure the [NVIDIA Container Toolkit](https://docs.nvidia.com/datacenter/cloud-native/container-toolkit/latest/install-guide.html)
is installed on your host.

## Adding / changing services (dynamic setup)

`services.conf` is the single source of truth. `setup.sh` reads it and
regenerates `local/frpc.ini` and `vps/docker-compose.yml` automatically.

**Format:**
```
name : local_port : remote_port : local_host (optional)
```

| `local_host` value | When to use |
|--------------------|-------------|
| *(omitted)*        | Service runs as a native process on your machine |
| docker service name (e.g. `ollama`) | Service is in the same `local/docker-compose.yml` |

**Example — add Open WebUI and a custom API:**
```
# services.conf
ollama       : 11434 : 11434 : ollama
open-webui   : 3000  : 3000
my-api       : 8080  : 8080
```

Then run:
```bash
./setup.sh
```

Output tells you exactly which ports to open on the VPS. Then reload:
```bash
# Local
docker compose -f local/docker-compose.yml up -d --force-recreate frpc

# VPS (copy updated compose + restart)
scp -r vps/ root@<VPS_IP>:~/proxy/
ssh root@<VPS_IP> 'cd ~/proxy/vps && docker compose up -d'
```

> `local/frpc.ini` and `vps/docker-compose.yml` are auto-generated — don't
> edit them by hand.

## Dashboard

Visit `http://<YOUR_VPS_IP>:7500` — log in with `admin` /
`DASHBOARD_PWD` to see connected tunnels and traffic stats.

## Security notes

- The FRP tunnel itself is authenticated with `FRP_TOKEN` but traffic is **not
  encrypted end-to-end** by default. For production use, put Nginx + TLS in
  front of port 11434 on the VPS, or enable `tls_enable = true` in frps/frpc.
- Never commit `.env` files — `.gitignore` already excludes them.
